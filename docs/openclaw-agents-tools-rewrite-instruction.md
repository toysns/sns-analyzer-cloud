# OpenClaw AGENTS.md / TOOLS.md 書き換え指示

## お前にやってほしいこと

AGENTS.md と TOOLS.md を、RULES.md の2層構造（哲学 + ハードルール4つ）と整合させろ。行動ルールが紛れ込んでいる箇所を削除・修正しろ。

---

## AGENTS.md の修正

### このファイルの役割を守れ
AGENTS.md は「起動手順とメモリ管理」だけ。行動ルールを書くな。

### 削除するもの

#### 「グループチャット」セクション → 全削除
- 「Toyの代弁者にならない」「全メッセージに反応しない」「価値を追加できるときだけ発言」は行動ルール
- これらは RULES.md 第1層「お前はToyの拡張である」から導出できる。もう1人のToyなら、グループチャットでの振る舞いを箇条書きで確認したりしない
- リアクションのルールも同様。自然にやれ

#### 「長時間タスクの進捗報告」セクション → 全削除
- 「10分ごとに報告」は運用ルール。起動手順ではない
- 拡張なら、必要なタイミングで必要な粒度で報告する。固定間隔の指定は不要

### 修正するもの

#### メモリ管理の「書き込みルール」から以下の1行を削除：
> 同じ指摘を2回受けた → RULES.md に恒久ルールとして書く

これは危険。RULES.md は2層構造で完結させた。OpenClawが自分でルールを追加していく仕組みを残すと、ルールが増殖して元に戻る。同じ指摘を2回受けたら MEMORY.md に教訓として書け。RULES.md には書くな。

## TOOLS.md の修正

### このファイルの役割を守れ
TOOLS.md は「ツール固有の技術情報」だけ。承認ルールを書くな。

### 「承認レベル一覧」セクション → 全削除

理由：RULES.md 第2層で承認が必要な行動は4つだけと定義した。

1. 外部への発信
2. 金が動くこと
3. 消えたら戻せないこと
4. 公開されること

TOOLS.md に独自の承認テーブルがあると、どちらが正なのか分からなくなる。承認の判断基準は RULES.md の4つだけ。ツールごとに個別の承認テーブルは持つな。

例えば今のTOOLS.mdでは「削除 → 実況報告してから実行」になっているが、RULES.md のハードルール3は「消えたら戻せないこと → 事前に確認を取れ」。矛盾している。RULES.md が正。

### 残すもの
以下はツール固有の技術情報だから残していい：
- ブラウザプロファイル（openclaw / chrome の使い分け）
- claude -p 操作メモ
- X（旧Twitter）アクセス手順
- 既知の制約

## 書き換え時の注意

- RULES.md、SOUL.md、USER.md には手を入れるな
- 「これはルールか？手順か？」で迷ったら、ルール側（RULES.md に属するもの）として削除しろ
- 承認の判断基準は RULES.md の4つだけ。他のファイルに承認ルールを分散させるな
- 書き換えたら diff を見せてくれ
