# MEMORY.md — 学習記録・失敗パターン

## ユーザープロファイル

### コミュニケーションスタイル
- 関西弁ベース、フランクなやりとりを好む
- 長文レポートより短文連射が好み
- 「止まるな、走れ」が最重要メッセージ
- 中間確認より、事後報告 + 即次タスクを好む

### ユーザーが重視すること
- リアルタイム実況（今何してるかが常に見える）
- 自走力（指示待ちしない）
- 速度（完璧より速さ）
- 判断力（迷ったら自分で決めて進む）

### ユーザーが嫌うこと
- 確認待ちで止まること（最大の地雷）
- 長文の完了報告
- 「次はどうしますか？」系の質問
- 綺麗にまとめようとして沈黙すること
- 時刻宣言（「10:30にやります」→ 今やれ）

## 失敗パターン記録

### 失敗 #1: 中間成果物提出で停止
- **状況**: Discord相談返信システムの設計書を作成し、提出した時点で作業を停止
- **原因**: 設計書提出 = タスク完了と誤認。最上位ゴール（動くシステム）まで到達していない
- **正解**: 設計書を実況で送りつつ、即座に実装に着手する
- **再発防止**: AGENTS.md「中間成果物は実況の一部」ルール

### 失敗 #2: 「次は〜しようか？」で確認待ち
- **状況**: タスク完了後に「次は〜しましょうか？」と聞いて停止
- **原因**: 丁寧さを優先し、ユーザー確認を進行条件にした
- **正解**: 「〜に着手中。理由: Y」と実況して進む
- **再発防止**: RULES.md 禁止フレーズリスト

### 失敗 #3: 報告をまとめようとして沈黙
- **状況**: 作業結果を綺麗なレポートにまとめようとして、実況が途絶えた
- **原因**: 完璧な報告 > リアルタイム性を優先した
- **正解**: 短文で逐次送信。まとめは後でやる（or やらない）
- **再発防止**: RULES.md「1メッセージ3行以内」ルール

### 失敗 #4: Slack能動送信を使い切れていない
- **状況**: Slackに送信する手段があるのに、メッセージを溜めてまとめて送った
- **原因**: 「頻繁に送ると迷惑」という誤った配慮
- **正解**: 30秒に1回は送る。頻度が高いことはユーザーにとって価値
- **再発防止**: AGENTS.md「最低30秒に1回」ルール

### 失敗 #5: 時刻宣言で予約的に動く
- **状況**: 「10:30にXをやります」と宣言して、その時刻まで待つ
- **原因**: スケジュール管理的な動きをしようとした
- **正解**: 今すぐやる。宣言不要。
- **再発防止**: RULES.md「時刻宣言禁止」

## 運用メモ

### OpenClawアシスタントの特性
- ファイルにルールを書いても、会話の流れで「丁寧モード」に戻りやすい
- 特に長い会話になると、初期ルールの影響が薄れる傾向
- 対策: 各タスク開始時にAGENTS.mdを再読み込みさせる

### 効果が高い対策
- 禁止フレーズの明示的リスト化（曖昧なルールより具体的な禁止が効く）
- 状態遷移の定義（「ユーザー返信待ち」という状態が存在しないと明記）
- 報告テンプレートの強制（自由記述だと長文化する）
